# Исправление системы друзей

## Проблемы которые были исправлены

### 1. ❌ Кнопка "Принять запрос" была неактивной (серой)
**Проблема**: В поиске пользователей кнопка "Принять запрос" была disabled для статуса `pending_received`

**Решение**: 
- Убрали `pending_received` из условия disabled
- Теперь кнопка активна и позволяет принять запрос прямо из поиска
- При клике система автоматически находит входящий запрос и принимает его

**Файл**: `frontend/src/app/friends/page.tsx`

### 2. ❌ Входящие запросы не отображались
**Проблема**: Backend корректно возвращал входящие запросы, но они могли не отображаться из-за проблем с обновлением состояния

**Решение**:
- Добавлен вызов `fetchData()` после принятия запроса в поиске
- Это обновляет и список друзей, и входящие запросы одновременно

**Файл**: `frontend/src/app/friends/page.tsx`

### 3. ❌ Подписки (followers/following) не обновлялись
**Проблема**: Система друзей и система подписок (follow) были разделены. При добавлении в друзья счётчики подписчиков не менялись.

**Решение**:
- При принятии запроса в друзья автоматически создаются **взаимные подписки** (follow)
- При удалении из друзей удаляются и взаимные подписки
- Счётчики `followersCount` и `followingCount` обновляются автоматически

**Изменения**:
- `acceptFriendRequest()` - теперь создаёт follow отношения
- `removeFriend()` - удаляет follow отношения
- `sendFriendRequest()` - создаёт follow при автопринятии (когда оба отправили запросы)

**Файл**: `backend/src/modules/friends/friends.service.ts`

### 4. ❌ Чат не работал после добавления в друзья
**Проблема**: Chat rooms создавались, но могли не отображаться из-за проблем с relations

**Решение**:
- Backend уже корректно проверяет дружбу перед созданием чата
- Chat rooms автоматически создаются при первом сообщении
- Система корректно фильтрует комнаты где оба пользователя существуют

**Файл**: `backend/src/modules/chat/chat.service.ts` (без изменений - уже работал корректно)

## Как теперь работает система

### Поток добавления в друзья:

1. **Пользователь A отправляет запрос пользователю B**
   - Создаётся FriendRequest со статусом `pending`
   - Пользователь B видит запрос во вкладке "Входящие"

2. **Пользователь B принимает запрос**
   - Обновляется статус FriendRequest на `accepted`
   - Создаётся Friendship (A ↔ B)
   - **НОВОЕ**: Создаются два Follow отношения:
     - A → B (A подписывается на B)
     - B → A (B подписывается на A)
   - Увеличиваются счётчики подписок у обоих
   - ChatRoom будет создана при первом сообщении

3. **Удаление из друзей**
   - Удаляется Friendship
   - **НОВОЕ**: Удаляются оба Follow отношения
   - Уменьшаются счётчики подписок
   - ChatRoom остаётся, но новые сообщения отправлять нельзя

### Интеграция Follow + Friends:

```
ДРУЗЬЯ = взаимные подписки (оба follow друг друга)
ПОДПИСЧИК = односторонняя подписка (только один follow)
```

Теперь:
- Друзья ВСЕГДА имеют взаимные follows
- followersCount и followingCount корректно отображают количество подписок
- Система friends и follow интегрированы

## Технические детали

### База данных:
- `FriendRequest` - запросы в друзья (pending/accepted/rejected)
- `Friendship` - таблица дружбы (user1_id, user2_id где user1_id < user2_id)
- `Follow` - таблица подписок (follower_id, following_id)
- `ChatRoom` - комнаты чата (user1_id, user2_id где user1_id < user2_id)

### Каскадное удаление:
Все связанные сущности удаляются при удалении пользователя благодаря `onDelete: 'CASCADE'`

### Транзакции:
- `sendFriendRequest()` использует transaction с pessimistic lock
- `getOrCreateRoom()` использует transaction с pessimistic lock
- Предотвращает race conditions при одновременных операциях

### Индексы:
Все таблицы имеют правильные индексы для быстрого поиска:
- FriendRequest: `[senderId, receiverId, status]`
- Friendship: `[user1Id, user2Id]` (unique)
- Follow: `[followerId, followingId]` (unique)
- ChatRoom: `[user1Id, user2Id]` (unique)
